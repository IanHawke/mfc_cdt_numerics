Finite Differencing Schemes

Finite Differencing

Approximate advection equation

\[ \frac{\partial \phi}{\partial t} + \symbf{u} \cdot \symbf{\nabla} \phi = 0 \]

by eg \(\partial_x \to\) algebraic approximation.

Three standard approximations:

\[ \left. \frac{\partial \phi}{\partial x} \right|_{x = x_j} \simeq \begin{cases} \frac{\phi_{j+1} - \phi_{j}}{\Delta x} & \text{Forward} \\ \frac{\phi_{j} - \phi_{j-1}}{\Delta x} & \text{Backward} \\ \frac{\phi_{j+1} - \phi_{j-1}}{2 \, \Delta x} & \text{Centred} \end{cases} \]

We are trying to numerically approximate the solution to PDEs. Our first approach is finite differencing: introduce a grid and replace any partial derivative at that point with an algebraic approximation. We can then rearrange the resulting algebraic equation to get the unknown quantities in terms of known data.

There are many approximations, and some are useful some of the time. The three standard ones are listed here. As we have to approximate both derivatives in time and derivatives in space, we have nine possible methods just from these approximations alone. We looked at one - FTBS - in the last lecture, but there are qualitative differences that are important to consider.

Order of Accuracy

To check the approximation, Taylor expand:

\[ \phi_{j \pm 1} = \phi_j \pm \Delta x \, \left. \frac{\partial \phi}{\partial x} \right|_{x = x_j} + \frac{\Delta x}{2} \left. \frac{\partial^2 \phi}{\partial x^2} \right|_{x = x_j} \pm \frac{\Delta x^2}{6} \left. \frac{\partial^3 \phi}{\partial x^3} \right|_{x = x_j} + \mathcal{O}(\Delta x^3) \, . \]

Thus

\[ \begin{aligned} \text{Forward:} && \frac{\phi_{j+1} - \phi_{j}}{\Delta x} &= \left. \frac{\partial \phi}{\partial x} \right|_{x = x_j} + \mathcal{O}(\Delta x) \, , \\ \text{Backward:} && \frac{\phi_{j} - \phi_{j-1}}{\Delta x} &= \left. \frac{\partial \phi}{\partial x} \right|_{x = x_j} + \mathcal{O}(\Delta x) \, , \\ \text{Centred:} && \frac{\phi_{j+1} - \phi_{j-1}}{2 \,\Delta x} &= \left. \frac{\partial \phi}{\partial x} \right|_{x = x_j} + \mathcal{O}(\Delta x^2) \, . \end{aligned} \]

So far we have treated the finite difference approximations as given, or as coming from approximations using a picture. However, we want to do better than that. We want to check that the approximations are correct, and in what sense, and also which ones are better. To do this we use Taylor expansions.

The Taylor expansion is done in terms of the grid spacing between neighbouring grid points, so assumes that is small. It also expresses the value at the neighbouring grid points in terms of derivatives of the function at the grid point, so assumes that those derivatives exist. This analysis (and indeed the finite differencing approximation in general) runs into trouble when these assumptions are broken.

By substituting the expansions into the finite difference approximations we see that all of them are “correct” in the sense that the leading order term is the derivative that we want, and the higher order terms will all vanish in the limit as the grid spacing goes to zero. This is often called the continuum limit.

We also see that both the forward and backward differencing approximations essentially give the same result. The leading order error is proportional to the grid spacing. It is also, in both cases, proportional to the second derivative of the function at the point. The only difference between the approximations is the sign of this leading order error.

However, the centred differencing approximation is different. The leading order error is proportional to the square of the grid spacing. As the grid spacing is assumed small, this would appear to be much more accurate than the alternatives. However, it uses information from both neighbours, which does not fit with the notion of characteristics.

Three Centred schemes

\[ \text{FTCS}: \quad \quad {\color{red}{\phi^{n+1}_j}} = {\color{green}{\phi^{n}_{j}}} - \frac{u \, \Delta t}{2 \, \Delta x} \left( {\color{green}{\phi^{n}_{j+1}}} - {\color{green}{\phi^{n}_{j-1}}} \right) \, . \]

Explicit.

\[ \text{BTCS}: \quad \quad {\color{red}{\phi^{n+1}_j}} = {\color{green}{\phi^{n}_{j}}} - \frac{u \, \Delta t}{2 \, \Delta x} \left( {\color{red}{\phi^{n+1}_{j+1}}} - {\color{red}{\phi^{n+1}_{j-1}}} \right) \, . \]

Implicit.

\[ \text{CTCS}: \quad \quad {\color{red}{\phi^{n+1}_j}} = {\color{green}{\phi^{n-1}_{j}}} - \frac{u \, \Delta t}{2 \, \Delta x} \left( {\color{blue}{\phi^{n}_{j+1}}} - {\color{blue}{\phi^{n}_{j-1}}} \right) \, . \]

Explicit, multiple levels needed.

As centred differencing seems to have advantages, let us look at the three schemes we can construct. In each case we follow the same procedure as for FTBS: introduce a grid, at each grid point evaluate the PDE with partial derivatives replaced with finite difference approximations, and then re-arrange the algebraic expression to isolate the unknown data.

The FTCS algorithm looks exactly like the FTBS algorithm. After re-arrangement we have exactly one unknown value (in red), specified in terms of known values (in green). The practical implementation difference to FTBS is small (aside from the precise values used, we need to modify the algorithm at boundaries in a different way).

The BTCS algorithm is different. Multiple values in the expression are unknown. This makes the algorithm implicit. It is still possible to solve the algorithm as we will see, as the collection of equations at all grid points forms a set of simultaneous linear algebraic equations that can be solved. As should be expected, we will use a matrix form to express this.

CTCS is different again. If we assume that the nth timestep is known then (as long as n is bigger than zero) the (n-1)th timestep is also known. Therefore the values in blue and the values in green on the right hand side are all known, and the algorithm is explicit. There are two points, however. First, the algorithm will be more expensive (in terms of memory usage) as we need to keep more information stored (the data at two past times). Second, the mathematical formulation of the PDE only gives us initial data at one time. In general, if using centred time approximations, we have to use a different method for enough timesteps to give approximations for the early times.

FTCS results

import numpy as np
u = 1
nx = 100
cfl = 0.8
nsteps = 100
d = [0, 1]
dx = (d[1] - d[0]) / nx
dt = cfl * dx / abs(u)
x, dx = np.linspace(d[0]+dx/2-dx, 
                    d[1]-dx/2+dx, 
                    nx+2, 
                    retstep=True)
phi = np.sin(2*np.pi*x)**4
for n in range(nsteps+1):
    phi[1:nx+1] -= u*dt/2/dx*(
        phi[2:nx+2]-phi[0:nx])
    phi[0] = phi[nx+1]
    phi[-1] = phi[1]

We apply the FTCS scheme to the advection equation with constant advection velocity one on the periodic unit domain. This means each unit time the solution will advect around the domain back to where it started. We have a sketch code and the results for some early times.

The code is nearly identical to that for FTBS. In the setup section the only difference is the number of steps taken, for reasons that will soon be obvious. The setup of the grid spacings and the grid are identical, as is the initial data. The key difference is in the update step, particularly lines 15 and 16, where the factor two and the change in the points used is needed to implement FTCS. The boundary conditions are identical, as we are using the ghost zone approach.

The results show that the scheme advects the profile, and the speed of propagation is not obviously terrible. However, the profile grows in amplitude and introduces new oscillations as it travels. This growth in the amplitude leads to the approximation “blowing up”, and the result is unstable. We will make this more precise later.

BTCS implementation

\[ \text{BTCS}: \quad \quad {\color{red}{\phi^{n+1}_j}} + \frac{u \, \Delta t}{2 \, \Delta x} \left( {\color{red}{\phi^{n+1}_{j+1}}} - {\color{red}{\phi^{n+1}_{j-1}}} \right) = {\color{green}{\phi^{n}_{j}}} \, . \]

Gather all equations as

\[ M {\color{red}{\symbf{u}^{n+1}}} = {\color{green}{\symbf{u}^n}} \, . \]

Matrix

\[ M = \begin{pmatrix} \dots & \dots & \dots & \dots & \dots & \dots & \dots \\ \dots & 0 & -\frac{u \, \Delta t}{2 \Delta x} & 1 & \frac{u \, \Delta t}{2 \Delta x} & 0 & \dots \\ \dots & \dots & \dots & \dots & \dots & \dots & \dots \end{pmatrix} \]

Matrix is sparse: important for practicality.

BTCS is fundamentally different to the explicit methods in that the update equation couples unknown values at different points. As the equation and scheme are both linear, this leads to a set of linear simultaneous equations. As there is one equation for each grid point (ignoring the issues with boundary conditions for now), this is solveable.

By writing all the unknown terms on the left-hand-side of the equation we can easily see how to set up a matrix equation. Because of the long-standing link between computing and linear algebra, there are many standard ways to rapidly solve such linear systems.

However, it is important for practical purposes to note that the matrix is sparse: a typical row of the matrix will contain only three non-zero entries. Therefore the total number of non-zero entries is roughly three times the number of grid points. If the matrix were dense, it would go as the number of grid points squared. For large grids in the continuum limit the difference in memory usage and performance is massive.

BTCS results

from scipy import sparse
from sparse.linalg import spsolve
A = sparse.lil_array((nx, nx))
for j in range(1, nx-1):
    A[j, j-1] = -u * dt / (2 * dx)
    A[j, j  ] = 1
    A[j, j+1] =  u * dt / (2 * dx)
A[0, 0 ] = 1
A[0, 1 ] =  u * dt / (2 * dx)
A[0, -1] = -u * dt / (2 * dx)
A[-1, 0 ] =  u * dt / (2 * dx)
A[-1, -2] = -u * dt / (2 * dx)
A[-1, -1] = 1
M = A.tocsr()
for n in range(nsteps+1):
    phi[1:nx+1] = spsolve(M, 
                phi[1:nx+1])
    phi[0] = phi[nx+1]
    phi[-1] = phi[1]

We apply the BTCS scheme to the advection equation with constant advection velocity one on the periodic unit domain. This means each unit time the solution will advect around the domain back to where it started. We have a sketch code and the results for some early times.

The code is substantially different to previous methods. We omit the standard setup of the grid and grid spacings. Instead we include the setup of the matrix. We use scipy’s sparse matrix approach, which is the only way to work when the grid gets large. This relies on different formats; an easy-to-use “list of lists” approach when setting up the coefficients, and then a more compact format when actually solving the method.

The results show that the scheme advects the profile, and the speed of propagation is reasonable. Unlike FTCS the results are clearly stable. However, the damping of the amplitude is substantial, not unlike FTBS.

CTCS results

phi = np.sin(2*np.pi*x)**4
phi_p = np.sin(2*np.pi*(x+u*dt))**4
for n in range(nsteps+1):
    phi_n = np.copy(phi)
    phi_n[1:nx+1] = phi_p[1:nx+1]-(
        u*dt/2/dx*(
        phi[2:nx+2]-phi[0:nx]))
    phi_n[0] = phi_n[nx+1]
    phi_n[-1] = phi_n[1]
    phi_p = np.copy(phi)
    phi = np.copy(phi_n)

We apply the CTCS scheme to the advection equation with constant advection velocity one on the periodic unit domain. This means each unit time the solution will advect around the domain back to where it started. We have a sketch code and the results for some early times.

Again we eliminate the setup code. However, we now need to set two sets of initial data. In this case we take advantage of the exact solution; in general we would need to use, say, FTBS for one step.

The update is then very similar. The key difference is that it uses known data from two different times. Once the boundaries are set, we then need to update what the “previous time” data points to.

The actual results are easily the best we have seen. The data is now evolved for five times longer than the FTBS or BTCS results, and yet the amplitude is still captured incredibly well. We start to see some distortion in the profile, howeer.

Summary

  • There are a zoo of finite difference approaches.
  • Taylor expansion is a key tool.
  • Order of accuracy is a useful metric, but incomplete.
  • Explicit methods “easier” than implicit.
  • Stability issues with some methods.

The key takeaway from these different methods is that the key things we will care about - stability and accuracy - are not always obviously linked to our choices of differencing method.
